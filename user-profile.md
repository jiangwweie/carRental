# 租车应用（Car Rental）用户画像

> 个人自营租车业务，面向旅游场景用户

## 1. 目标用户画像

| 维度 | 描述 |
|---|---|
| **核心用户群** | 25-45 岁自驾游爱好者、家庭出游人群、商务差旅人士 |
| **使用场景** | 旅游度假租车、机场/高铁站接送租车、短期商务出行租车、周末短途自驾 |
| **使用频率** | 低频高客单价（年均 1-4 次租车，单次 1-7 天） |
| **决策因素** | 价格透明度、车型丰富度、取还车便利性、保险保障、车况 |
| **付费意愿** | 中等偏上，愿意为便利性、车况好、服务省心支付溢价 |
| **设备偏好** | 移动端优先（微信内 H5/小程序），少数场景使用 PC 端预订 |
| **地域分布** | 旅游城市及周边（海南、云南、川西、新疆等热门自驾目的地） |

## 2. 用户分层

| 分层 | 占比 | 特征 | 核心诉求 |
|---|---|---|---|
| **价格敏感型** | 40% | 学生、年轻白领，预算有限，比价行为明显 | 低价、优惠券、透明无隐形消费 |
| **品质优先型** | 35% | 中产家庭、商务人士，注重体验 | 好车况、保险齐全、取还车便捷、客服响应快 |
| **便利驱动型** | 25% | 行程紧张、临时需求，追求效率 | 快速预订流程、就近取车、灵活改期 |

## 4. 用户体验痛点与解决方案

| 用户痛点 | 影响 | 解决方案 |
|---|---|---|
| 隐藏费用不透明 | 信任度低，差评率高 | 价格明细前置展示，所有费用在确认页一次性列明 |
| 取还车纠纷 | 客诉多，运营成本高 | 取还车拍照存档 + 电子签名确认，全程留痕 |
| 预订流程繁琐 | 转化率低 | 简化为 3 步：选车 → 确认 → 支付 |
| 临时取消损失大 | 用户犹豫，弃单率高 | 阶梯式取消政策，提前展示退款规则 |
| 车况与描述不符 | 信任危机 | 实拍图 + 车型参数标准化，支持到店验车退换 |
| 客服响应慢 | 紧急场景体验差 | 微信内嵌客服入口，关键节点自动通知 |

## 5. 技术栈匹配度分析

| 技术选型 | 匹配理由 | 风险点 |
|---|---|---|
| **React 18+ / TypeScript 5+** | 类型安全、组件复用、大型表单场景友好 | 学习曲线，需规范 TS 使用 |
| **TailwindCSS 3+** | 快速构建响应式 UI，移动端适配 | 类名膨胀，需配合组件库约束 |
| **Zustand** | 轻量状态管理，适合中等规模应用 | 复杂场景需规划 store 拆分 |
| **Vite** | 极速 HMR，开发体验好 | 生态较新，插件兼容性需验证 |
| **FastAPI** | 高性能、自动文档、类型提示，适合 RESTful API | 异步生态成熟度不如 Django |
| **SQLAlchemy 2.0** | 成熟的 ORM，灵活查询构建 | 2.0 新语法需团队统一规范 |
| **Alembic** | 数据库版本管理，与 SQLAlchemy 无缝集成 | 迁移脚本需 code review |
| **PostgreSQL** | 事务一致性、JSON 支持、地理查询（未来扩展） | 运维成本高于 SQLite |
| **Docker + Compose** | 环境一致性，一键部署 | 镜像优化和编排复杂度 |
| **Nginx** | 反向代理、静态资源、SSL 终止 | 配置需规范化管理 |
| **微信 JSAPI 支付** | 微信支付覆盖率广，用户信任 | 商户号申请周期，签名调试 |
| **对象存储（COS/OSS）** | 图片/视频存储，CDN 加速 | 费用控制，防盗链策略 |

## 10. 项目优缺点评估

### 优点

- **业务聚焦**：租车场景明确，功能边界清晰，MVP 范围可控
- **技术栈成熟**：前后端选型均为当前主流生态，社区活跃，Agent 资源丰富
- **架构简洁**：单体 + 容器化部署，运维复杂度低，适合小团队
- **扩展性强**：PostgreSQL + FastAPI 组合为后续功能扩展（多门店、会员体系、推荐算法）留足空间
- **契约驱动**：三阶段工作流确保前后端并行开发不阻塞，10 人 Agent Team 分工明确

### 风险与挑战

- **支付对接**：微信 JSAPI 支付需要在微信环境内调试，商户号申请和签名调试耗时
- **低频场景**：用户粘性低，需依赖营销和口碑驱动增长
- **合规风险**：租车行业涉及保险、交通法规，用户协议需专业法律审核
- **图片存储成本**：取还车拍照存档会产生大量图片，需合理配置 COS/OSS 生命周期策略
- **节假日定价策略**：动态定价算法需要历史数据支撑，初期需人工配置 HolidayConfig
- **Agent 协作效率**：10 人 Agent Team 的契约管理复杂度高，接口变更需严格版本控制

### 改进建议

1. **MVP 聚焦**：首期只覆盖核心预订 → 支付 → 取还车闭环，评价/优惠券/多门店二期再做
2. **监控先行**：上线即配置基础监控（API 响应时间、错误率、支付成功率）
3. **灰度策略**：先服务 1-2 个取车点，跑通流程后再扩展
4. **文档沉淀**：API 文档（OpenAPI）、部署手册、运维手册在开发期同步编写
5. **契约锁定**：Phase 1 输出的 OpenAPI Spec 作为不可变契约，Phase 2 开发严格按契约执行
